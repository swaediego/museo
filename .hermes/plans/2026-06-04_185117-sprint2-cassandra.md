# Sprint 2 — Integración de Apache Cassandra (Históricos, Auditoría y Reportes)

> **Para Hermes:** Este plan se entrega al usuario como propuesta. La ejecución se delega task-by-task con subagent-driven-development cuando el usuario apruebe.

**Goal:** Sumar Apache Cassandra como microservicio de apoyo AP al stack políglota (PostgreSQL + MongoDB) para responder 3 consultas gerenciales (historial de precios, auditoría de ventas, bitácora de eventos) con persistencia asíncrona y desacoplada del core transaccional.

**Architecture:** Cassandra 5.0 como nuevo servicio en docker-compose. Spring Boot 3.4.4 + `spring-boot-starter-data-cassandra`. El flujo de venta/precio/cambio-estatus emite un `ApplicationEvent` ANTES de retornar al cliente; un `@TransactionalEventListener(phase = AFTER_COMMIT)` lo inserta en Cassandra. La transacción Postgres nunca se bloquea esperando a Cassandra (CAP: priorizamos Disponibilidad). Modelo de datos 100% query-driven: una tabla por consulta requerida, sin normalización.

**Tech Stack:** Apache Cassandra 5.0 · Spring Data Cassandra · Spring Application Events (publish/listen) · Docker Compose · CQL.

## 1. Modelo de Datos (Query-Driven, 3 tablas = 3 consultas)

Justificación: en Cassandra no hay JOIN. Cada tabla modela exactamente UNA consulta. Los datos se duplican entre tablas si dos consultas lo requieren, y eso está bien (es la regla, no la excepción).

### 1.1 `historial_precio_por_obra`
- **Consulta objetivo:** "Dame toda la historia de cambios de precio de la obra X ordenados del más reciente al más viejo"
- **Partition Key:** `id_relacional` (int) — agrupa toda la historia de UNA obra en una sola partición (lectura O(1))
- **Clustering Columns:** `fecha_cambio` (timestamp, **DESC**) — orden natural = más reciente primero; `id_evento` (uuid, ASC) — desempate único
- **Columnas:** `precio_anterior`, `precio_nuevo`, `motivo`, `usuario_admin`, `id_evento`
- **TTL:** opcional 5 años (auditoría fiscal), pero conservamos indefinido por ahora

```cql
CREATE TABLE museo_history.historial_precio_por_obra (
    id_relacional    int,
    fecha_cambio     timestamp,
    id_evento        uuid,
    precio_anterior  decimal,
    precio_nuevo     decimal,
    motivo           text,
    usuario_admin    text,
    PRIMARY KEY ((id_relacional), fecha_cambio, id_evento)
) WITH CLUSTERING ORDER BY (fecha_cambio DESC, id_evento ASC);
```

### 1.2 `auditoria_ventas_por_periodo`
- **Consulta objetivo:** "Resumen de facturación de YYYY-MM-DD a YYYY-MM-DD, ordenable por fecha"
- **Partition Key:** `periodo` (text formato "YYYY-MM") — la consulta siempre es por mes, y un mes cabe cómodo en una partición (cientos-miles de ventas/mes)
- **Clustering Columns:** `fecha_venta` (timestamp, DESC), `id_factura` (int, ASC) — desempate
- **Columnas:** `id_obra`, `id_comprador`, `monto`, `impuesto`, `metodo_pago`, `estatus_factura`

```cql
CREATE TABLE museo_history.auditoria_ventas_por_periodo (
    periodo          text,
    fecha_venta      timestamp,
    id_factura       int,
    id_obra          int,
    id_comprador     int,
    monto            decimal,
    impuesto         decimal,
    metodo_pago      text,
    estatus_factura  text,
    PRIMARY KEY ((periodo), fecha_venta, id_factura)
) WITH CLUSTERING ORDER BY (fecha_venta DESC, id_factura ASC);
```

### 1.3 `bitacora_eventos`
- **Consulta objetivo:** "Eventos del sistema en un rango de fechas, filtrables por tipo"
- **Partition Key:** `periodo_dia` (text "YYYY-MM-DD") — un día de eventos cabe en una partición
- **Clustering Columns:** `timestamp_evento` (timestamp, DESC), `tipo_evento` (text, ASC)
- **Columnas:** `id_entidad`, `tipo_entidad`, `detalle_json` (text JSON), `severidad` (INFO/WARN/ERROR), `usuario_origen`

```cql
CREATE TABLE museo_history.bitacora_eventos (
    periodo_dia       text,
    timestamp_evento  timestamp,
    tipo_evento       text,
    id_entidad        int,
    tipo_entidad      text,
    detalle_json      text,
    severidad         text,
    usuario_origen    text,
    PRIMARY KEY ((periodo_dia), timestamp_evento, tipo_evento)
) WITH CLUSTERING ORDER BY (timestamp_evento DESC, tipo_evento ASC);
```

### 1.4 Decisiones de partición (por qué así)
- **Hot partition avoidance:** el PK mensual/diario evita que una sola partición reciba millones de escrituras (problema clásico de Cassandra).
- **No usar idRelacional como PK en auditoría de ventas** porque una sola obra popular recibiría todas las escrituras del mes → hotspot. Por eso la PK es el período.
- **DESC en clustering** porque la consulta típica es "lo más reciente primero" (gerencial).
- **Sin secondary indexes** en esta primera versión: cada tabla ya cubre una consulta; cuando agreguemos filtros secundarios, lo haremos con tablas adicionales de "vista materializada" (Cassandra pattern estándar).

      interval: 15s
      timeout: 10s
      retries: 12
      start_period: 60s   # Cassandra tarda en arrancar
    networks:
      - default

  backend:
    # ... (existente) ...
    depends_on:
      db:
        condition: service_healthy
      mongodb:
        condition: service_started

      - SPRING_DATA_CASSANDRA_LOCAL_DATACENTER=dc1
      - SPRING_DATA_CASSANDRA_KEYSPACE_NAME=museo_history
      - SPRING_DATA_CASSANDRA_SCHEMA_ACTION=CREATE_IF_NOT_EXISTS

Y agregar al final del archivo:
```yaml
volumes:

  cassandra_data:
```

### 2.2 Por qué `CREATE_IF_NOT_EXISTS` en el backend
Spring Data Cassandra puede generar el schema de las entidades en el primer arranque. Esto evita ejecutar un `.cql` a mano y le da al estudiante un solo comando (`docker compose up -d --build`) para tener todo levantado. Las decisiones de modelado viven en las anotaciones de las entidades Java, no en archivos CQL sueltos (DRY).

### 3.4 Listener asíncrono (consumo)

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class CassandraHistoryListener {

    private final HistorialPrecioRepository precioRepo;
    private final AuditoriaVentaRepository   ventaRepo;
    private final BitacoraEventoRepository   bitacoraRepo;

    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onPrecioCambiado(PrecioActualizadoEvent e) {
        try {
            precioRepo.save(HistorialPrecio.builder()
                .idRelacional(e.idRelacional())
                .fechaCambio(e.fecha())
                .idEvento(UUID.randomUUID())
                .precioAnterior(e.precioAnterior())
                .precioNuevo(e.precioNuevo())
                .motivo(e.motivo())
                .usuarioAdmin(e.usuario())
                .build());
            bitacoraRepo.save(/* evento de bitácora "PRECIO_ACTUALIZADO" */);
        } catch (Exception ex) {
            log.error("No se pudo escribir historial de precio en Cassandra: {}", ex.getMessage());
            // NO relanzamos: la venta/actualización ya fue committeada.
        }
    }

    // onVentaRegistrada(...) y onEstatusCambiado(...) análogos
}
```

**Decisión clave:** el listener NO relanza la excepción. Si Cassandra está caída, Postgres sigue funcionando y el reporte gerencial simplemente llega tarde (consistencia eventual, justo lo que dice el requisito AP). El log queda para reconciliación manual.

## 5. Talking points para la defensa (por qué esto cumple el Sprint 2)

1. **CAP — Disponibilidad sobre Consistencia.** Si Cassandra está caída, el core transaccional (Postgres) sigue vendiendo; los reportes gerenciales se llenan cuando Cassandra vuelve. Eso es AP en acción. Postgres, en cambio, ante falla de réplica rechaza la lectura (CP), por eso las ventas van ahí.
2. **Query-driven modeling.** Las 3 tablas NO se parecen a `Art`/`Invoice` de Postgres. Cada una está modelada para UNA consulta específica: PK = período/cliente, clustering = orden natural. No hay JOIN, no hay normalización, hay desnormalización intencional.
3. **Asincronía desacoplada.** `ApplicationEvent` + `@TransactionalEventListener(AFTER_COMMIT)` garantiza que (a) el commit en Postgres se haya hecho antes de intentar Cassandra, y (b) la venta no espera a Cassandra. Si la escritura a Cassandra falla, el sistema no se cae.
4. **Append-only / inmutabilidad.** Ningún `UPDATE` ni `DELETE` sobre las 3 tablas en la lógica de la app. Cassandra es por naturaleza append-only con timestamps, lo que sirve perfectamente a la auditoría.
5. **Escalabilidad horizontal.** Si el museo crece, sumar otro nodo Cassandra al cluster es transparente: nueva entry en `contact-points`, sin cambios de schema. Postgres no escala así (single-writer por diseño), Mongo escala pero con sharding complejo. Cassandra escala horizontalmente con cero downtime.

## 7. Riesgos / Trade-offs / Preguntas abiertas

- **Riesgo (bajo):** el primer arranque de Cassandra tarda 30-60s. Si el backend arranca antes, va a tirar `Connection refused` y el listener va a loguear el error. El admin lo verá en logs, no rompe la app. Workaround: `start_period: 60s` en el healthcheck + `depends_on: condition: service_healthy`.
- **Riesgo (medio):** `CREATE_IF_NOT_EXISTS` crea el keyspace con `SimpleStrategy` y `replication_factor: 1` (default Spring). En dev está bien; en prod se cambia a `NetworkTopologyStrategy` con `dc1: 3`. Documentar esto en el README.
- **Pregunta abierta 1:** ¿querés que el admin pueda exportar los reportes a CSV/Excel? Es 30 min más de trabajo, lo dejo afuera hasta que confirmes.
- **Pregunta abierta 2:** ¿la bitácora incluye eventos de seguridad (login fallido, cambio de rol) o solo cambios de obras/estatus? El requisito dice "y eventos de seguridad", así que asumo ambos. Confirmame.
- **Pregunta abierta 3:** ¿querés que el listener publique también a Kafka/RabbitMQ para que sea 100% event-driven, o con `ApplicationEvent` in-process alcanza? Para la defensa alcanza con in-process (es lo estándar de Spring); Kafka suma complejidad sin beneficio funcional.
